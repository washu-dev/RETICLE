RETICLE

A pipeline that turns 2,157 mutually-incomparable public CRISPR screens into one harmonized, queryable similarity network — so a gene's behaviour in one screen can be read in the context of every other screen that ever measured it.

2,157
CRISPR screens (1,952 human · 205 mouse)
28.2 M
harmonized gene-level scores
542,698
stored screen-pair correlations

The problem

Every public screen is an island. One reports a CERES score near −2, another a BAGEL Bayes Factor of 0–10, a third a STARS score of 0–130. They use knockout, CRISPRi, or CRISPRa; negative, positive, or phenotype selection; whole-genome or focused libraries. The same number means totally different biology across screens — so they cannot be compared out of the box.

Phase 1 · Harmonization DONE

Map every screen into one common coordinate system where the same number means the same biology. Script: script/harmonize_scores.py.

The unified axis

Every gene is placed on a single signed axis, always in the loss-of-function frame:

−1 · essential (depletes)0 · no effect+1 · protective (enriches)
HARMONIZED_SCORE = S_raw × perturbation_multiplier

Perturbation multiplier

  • Knockout / CRISPRi / CRISPRn → +1
  • Activation (CRISPRa) → −1 — if activating a gene is protective, then losing it is deleterious, so we invert to stay in the loss-of-function frame.

How each raw score becomes S_raw

The meaning of a SCORE column is resolved from an explicit registry of the real SCORE.k_TYPE strings (≈58 of them), each assigned one role:

Priority: a directional column always wins over a significance column; a positive/negative pair (e.g. MAGeCK pos + neg p-values) is differenced; only unsigned significance falls back to the selection sign (Negative → −1, Positive → +1; ambiguous → +1, tagged [AMBIGUOUS_SELECTION]).

Two scale-free metrics — and which one Phase 3 uses

PERCENTILE_SCORE ∈ [−1,+1]  used by correlation

Percentile = 2·(rank / maxrank) − 1

Average-rank of HARMONIZED_SCORE over measured genes. This is the value Phase 3 correlates — because it is a pure rank, a Pearson correlation of two screens' percentile vectors is Spearman's ρ, and it neutralizes every scale difference (STARS 134 and a fold-change 6e-5 both map near +1).

ROBUST_Z_SCORE  stored, not used in correlation

Z = (S − median) / (1.4826 · MAD)

Median/MAD standardization. It is deliberately not used for correlation (it keeps each screen's raw scale, so it is not cross-comparable like a rank). Its job is to preserve effect magnitude for magnitude-aware uses — ranking how strong a hit is within a screen, tail/effect-size thresholds — which a flat percentile rank throws away. Both are stored so each analysis can pick the right one.

Missing data & provenance

  • Missing value (-) → NaN, excluded from ranking, stored as NULL percentile/z (never dumped at 0.0).
  • COVERAGE_TYPE from FULL_SIZE_AVAILABLE: FULL → continuous comparison, HIT_ONLY → binary.
  • Each screen records SCORE_BASIS (which path produced S_raw) and IS_DIRECTIONAL.
Where the sign came from (IS_DIRECTIONAL):
directional metric — 1,858 (86%)
selection type — 299 (14%)
Outputs: the harmonized_scores table (28.2 M rows) + three provenance columns on screen_metadata. Validated by validate_harmonization.py: core-essential genes (POLR2A, RPL3, PSMB3…) must sit at percentile ≈ −1 in every viability screen, regardless of method — currently PASS.

Phase 2 · Metadata Curation FULLY AUTOMATED

Two jobs. (1) A normalized label for every screen, by deterministic rules (script/llm_metadata_extractor.py). (2) The LLM's one real job — assigning a score direction to the minority of screens Phase 1 could not sign (script/directionality_mapper.py → apply_directionality.py).

The rule / LLM split

Most labels are already structured in BioGRID, so they are computed deterministically by rules (instant, confidence 1.0). The LLM is reserved for the one thing rules genuinely cannot get from structured fields — directionality (which sign of a screen's score means "knockout helps" vs "gene is essential"). There is no human-review step; confidence is for downstream filtering, not manual sign-off.

RULE normalized labels

  • screen_type ← METHODOLOGY + LIBRARY_TYPE + ENZYME
  • selection_method ← SCREEN_TYPE
  • coverage_type ← FULL_SIZE + FULL_SIZE_AVAILABLE

LLM gpt-4o · directionality

  • Reads BioGRID free text → the sign of an ambiguous screen's score.
  • Returns mode + sign + a verbatim evidence quote + confidence; schema-validated JSON.
  • Output frozen to directionality_overrides.json — a deterministic input so Phase 1 stays a pure function.

When is a screen "ambiguous"? (what triggers the LLM)

Phase 1 already signs 86% of screens with no LLM, straight from the score's own type (a log2FC or CERES value carries its direction in its sign). A screen is handed to the LLM only when both of these are true:

  • its only usable column is an unsigned significance metric — a p-value, FDR, STARS, CasTLE, RSA, or RRA/MaGeCK p — a number that says "this gene changed significantly" but not whether it went up or down; and
  • SCREEN_TYPE can't supply the missing sign — it is Phenotype Screen (a FACS/reporter sort with no built-in enrich/deplete axis) or Positive and Negative Selection (both directions at once).

That is 100 screens (AMBIGUOUS). A further 11 (UNRESOLVED) have no usable effect column at all (only read counts / sgRNA number / rank) → no continuous score is possible, so they are routed straight to Phase 3's BINARY mode. The polarity the rules lack is, for 96% of screens, spelled out in plain English in the curator NOTES — e.g. screen 89: "negative MaGeCK score = cell-essential genes, their KO results in decreased proliferation." That sentence is exactly what the LLM mines.

What the LLM sees, and what it returns

The call is deterministic (temperature = 0), JSON mode. The system prompt fixes the unified axis (+1 = loss-of-function advantageous/enriched, −1 = gene essential/depleted), the three output modes, and the schema. The user message carries only BioGRID fields (no PubMed, no paper full text):

INPUT → model

SYSTEM: axis (+1/−1) · decide SINGLE | PAIR | UNDEFINED · quote the exact NOTES phrase · return ONLY JSON USER: (screen 89) METHODOLOGY: Knockout SCREEN_TYPE: Positive and Negative Selection PHENOTYPE: cell proliferation EXPERIMENTAL_SETUP: Timecourse SCORE COLUMN LAYOUT: SCORE.1=MaGeCK Score, SCORE.2=p-Value, SCORE.3=MaGeCK Score, SCORE.4=p-Value SIGNIFICANCE_CRITERIA: Score.1 (MaGeCK) <= … OR Score.3 (MaGeCK) <= … SCREEN_RATIONALE: Cell-essential & growth restrictive NOTES: "negative MaGeCK score = cell-essential genes, KO → decreased proliferation; positive MaGeCK score = growth restrictive, KO → increased…" (≤1500 ch)

OUTPUT ← model

One validated JSON object. SINGLE → a global sign; PAIR → which column is the +/− side; UNDEFINED → polarity not in the text.

{ "mode": "PAIR", "positive_column": "SCORE.3", # enriched / + "negative_column": "SCORE.1", # depleted / − "sign": null, "confidence": 1.0, "evidence": "negative MaGeCK score are cell-essential genes… positive… growth restrictive" }

Frozen into directionality_overrides.json with status, evidence, model, and prompt version for provenance and audit.

How the answer becomes a sign — and the safety net

  • SINGLE → HARMONIZED_SCORE = magnitude × sign. PAIR → signal(positive_column) − signal(negative_column). This is the final sign (perturbation already folded in), applied non-destructively in place by apply_directionality.py.
  • Confidence gate: confidence < 0.7 or UNDEFINED → needs_review, not applied.
  • Biological sanity veto: in a knockout viability screen, core-essential genes (POLR2A, RPL3, PSMB3…) must sit at the negative extreme. If an override lands them on the positive side, the call is rejected. This is not optional: gpt-4o-mini inverted screen 89 at confidence 1.0 — only the essential-gene check caught it, which is why the model was upgraded to gpt-4o.
  • Ground-truth override: where essential genes give a deterministic answer, that answer beats the LLM — 3 screens were auto-corrected this way (ANCHOR_SINGLE).

Calling the LLM (WashU AI gateway)

  • Auth: Azure AD client-credentials → bearer token (cached, auto-refreshed). No API key on the official endpoint.
  • Endpoint: https://api.openai.wustl.edu/models/v1/chat/completions — token only.
  • Network: WashU-only — must be on campus or VPN (off-network → APIM 403).
  • Robustness: retry+backoff on 429/5xx, JSON-mode validation, and a preflight ping that aborts a full run cleanly if the gateway is unreachable (so an offline run never writes garbage rows).
Outputs: (1) screen_metadata_curated — one rule-labeled row per screen (all 2,157), and (2) directionality_overrides.json — the frozen LLM sign decisions, applied back into harmonized_scores (71 screens re-signed, SCORE_BASIS = LLM_SINGLE / LLM_PAIR / ANCHOR_SINGLE).

Phase 3 · Correlation Engine DONE

For every comparable pair of screens, how similar are their gene-level results? Script: script/compute_correlations.py.

What may be compared

  • Same organism only (human/mouse share almost no gene symbols).
  • On the gene intersection — only genes both screens measured count; the overlap size is stored so a ρ can be weighted by how many genes back it.

Which score does a correlation use? (one screen has many scores)

Exactly one number per (screen, gene): the PERCENTILE_SCORE. A screen can report several raw columns — and some report both a positive (enrichment) and a negative (depletion) score for the same gene — but none of that reaches Phase 3. Phase 1 has already collapsed every raw column into a single signed HARMONIZED_SCORE per gene before correlation runs:

  • many columns → the resolver prefers the directional one, or, for a pos/neg pair, combines them as (positive − negative) on one axis;
  • positive vs negative score → not a choice made at correlation time. A gene significant on the depletion side lands near −1, significant on the enrichment side near +1; the percentile is its single net position on the loss-of-function axis;
  • that one value is ranked within the screen → PERCENTILE_SCORE ∈ [−1, +1].

So CONTINUOUS mode correlates two screens' percentile vectors — never two competing scores, never the raw columns, and not ROBUST_Z_SCORE (that one is kept only for magnitude/thresholding). The "which score?" question is fully settled upstream in Phase 1; Phase 3 just sees the one harmonized rank. (This is also why the directionality work above matters: a wrong sign in Phase 1 would put a gene at the wrong end of this axis and flip its correlations.)

Two modes, routed by COVERAGE_TYPE

CONTINUOUS — 314,963 (58%)
BINARY — 227,735 (42%)
both screens FULL → Spearman on percentiles ≥1 screen HIT_ONLY → Jaccard/Fisher on hit sets

CONTINUOUS — the math

A per-pair scipy.spearmanr over ~1.5 M pairs would be far too slow. Instead the engine builds two dense matrices over the gene universe (symbols measured in ≥2 FULL screens of that organism):

  • X (genes × screens) — the PERCENTILE_SCORE (0 placeholder where a gene is absent);
  • M (genes × screens) — a mask, 1 where the gene was measured, else 0.

All pairwise correlations over pairwise-complete genes then come from six matrix products — the mask M zeroes out absent genes in every term, so each pair is computed only on shared genes:

n = Mᵀ·M # shared-gene count per pair sx = Xᵀ·M # Σ x over genes shared with the other screen sy = sxᵀ sxy = Xᵀ·X sxx = (X∘X)ᵀ·M syy = sxxᵀ ρ = (n·sxy − sx·sy) / √( (n·sxx − sx²)·(n·syy − sy²) )

Because PERCENTILE_SCORE is a rank, this Pearson-of-percentiles is Spearman's ρ. The p-value is then derived analytically from (ρ, n), vectorised over all surviving pairs — no per-pair test:

t = ρ · √( (n − 2) / (1 − ρ²) ) p = 2 · P(Tn−2 > |t|)

Tail correlation (spearman_rho_tails): the same correlation but restricted to genes that are a real hit (|percentile| > 0.5) in either screen — since ~80% of genes sit in the noisy middle, this isolates the signal.

BINARY — the math

A HIT_ONLY screen has no genome-wide ranking, so similarity is computed on the author hit sets (IS_HIT = 1): H₁, H₂, drawn from a gene universe of size N.

Jaccard = |H₁ ∩ H₂| / |H₁ ∪ H₂| Fisher p = hypergeom.sf( s − 1, N, |H₁|, |H₂| ) # s = |H₁ ∩ H₂| = P(observing ≥ s shared hits by chance)

Both are kept because they disagree informatively: a pair can have tiny Jaccard (one screen has thousands of hits) yet a Fisher p of 1e-38 — a wildly non-random overlap Jaccard alone would hide. Trust Fisher when they disagree.

Storage thresholds (only pairs worth a row are kept)
MIN_OVERLAP_GENES500reject underpowered continuous pairs
RHO_MIN0.30minimum |ρ| to store
TAIL_CUTOFF0.50|percentile| above which a gene counts as a hit (tail)
MIN_TAIL_GENES20minimum tail genes to report ρ_tails
MIN_SHARED_HITS3minimum shared hits for a binary pair

Interpreting a row

  • ρ ≈ +0.6, p ≈ 0 — same biology; a shared hit is cross-validated.
  • ρ ≈ 0 — no relationship (a hit is treatment-specific, not generic fitness).
  • ρ < 0, significant — opposite rankings → antagonistic / context-dependent biology.
  • High Fisher, low Jaccard — strong asymmetric hit overlap; trust Fisher.
Output: the correlation_analysis table (542,698 rows), indexed for "give me the screens most similar to screen X, ranked."
