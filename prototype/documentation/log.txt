[2026-06-17]  Directionality Mapping —— 用 LLM 解决 Phase 1 无法定向的 screen
================================================================================

背景
----
- 原始设计文件 (context/ 下的 proposal / 架构图 / slides) 把 "directionality"
  列为 RETICLE 头号问题：BioGRID 只给 rank order，不给方向。架构图里有专门的
  "Directionality Mapping" + "Biological Sanity Checks" 两个 agent。
- Phase 1 (harmonize_scores.py) 用确定性规则解决了 86% 的 screen（score 自带方向，
  如 log2FC/CERES）。但有 111 个解决不了：
    * 100 个 AMBIGUOUS_SELECTION：只有无方向的显著性列（p-value/STARS/MaGeCK p 等），
      且 SCREEN_TYPE 模糊（Phenotype Screen / Positive and Negative Selection）
      -> 旧代码硬猜 sel=+1。
    *  11 个 UNRESOLVED：根本没有可用效应列（只有 read counts / sgRNA number / rank）。
- 实测发现 BioGRID 的 NOTES 字段（覆盖 96.3% 的 screen）常常明确写出极性，例如
  screen 89: "negative MaGeCK score = cell-essential genes, KO -> decreased
  proliferation"。这正是定向所需的信息。

决策
----
1. LLM 在本项目的职责从 "control_comparison"（无下游用途）重新定位到 "directionality"。
   - 删除 screen_metadata_curated 的 control_comparison / confidence_control 两列。
   - llm_metadata_extractor.py 改为纯规则版（screen_type/selection/coverage），不再调 LLM。
2. 架构：LLM 只跑一次，产出冻结的 JSON 产物 processed_data/directionality_overrides.json，
   被 harmonize / apply_directionality 当确定性输入读取 —— Phase 1 仍是纯函数，
   LLM 的非确定性被隔离在产物生成这一步。
3. 低置信度(<0.7)或 UNDEFINED -> needs_review，不自动应用（人工裁决）。
4. 锚点冲突的处理：先尝试升级模型（见下），仍冲突的用必需基因 ground truth 纠正。

新增 / 改动文件
--------------
- script/directionality_mapper.py   (新) LLM 方向裁决器 -> overrides.json
- script/apply_directionality.py    (新) 非破坏性就地应用 override + 必需基因否决闸
- script/harmonize_scores.py        抽出可复用函数(load_screen_df/add_rank_columns/
                                     apply_override/load_overrides) + override 钩子
                                     （仅当 overrides.json 存在时激活；重构经 screen 381
                                     逐基因验证行为不变）
- script/llm_metadata_extractor.py  改为纯规则版 (v4.0-rules-only)
- processed_data/directionality_overrides.json  (新) 冻结产物

统一轴定义（LLM 对齐的）
----------------------
  +1 = loss-of-function 有利 / 被选择群体富集（基因正常时促进坏结局）
  -1 = 基因必需 / 表型所需（KO 后耗竭，如核心必需基因）
  注：CRISPRa 等扰动方向由 LLM 推理 NOTES 时一并折入，输出即最终符号。

运行结果（gpt-4o, temperature=0）
-------------------------------
  目标: 111 (AMBIGUOUS=100, UNRESOLVED=11)
  token: prompt≈87,545 + completion≈8,880  (~96K, 几分钱)
  mapper 产出: auto=71  needs_review=29  binary_only=11
  apply 应用: 71 (LLM_SINGLE=64 + LLM_PAIR=4 + ANCHOR=3)
  全库 sanity check (validate_harmonization.py): PASS
     —— 11 个分析方法的核心必需基因全部稳在负端，方向修正未破坏既有数据。

关键事故与处理
-------------
1. gpt-4o-mini 把 screen 89 (PAIR) 判反，且 confidence=1.0。
   - 表现：把 positive_column/negative_column 弄反，必需基因 mean_pct = +0.274（应为负）。
   - 被 apply_directionality 的"必需基因否决闸"抓住（KO 增殖筛里必需基因落正端 = 方向反）。
   - 结论：置信度门控不足以拦错；生物学 sanity check 是必需的。
   - 处理：按用户决定升级到 gpt-4o。gpt-4o 把 89 判对 (+SCORE.3/-SCORE.1)，
     且 172(BMP激活子)= -1 与 173(BMP抑制子)= +1 符号相反，一致性正确。

2. gpt-4o 仍有 3 个误判：screen 1050 / 1052 / 1053。
   - 布局 {SCORE.1=MaGeCK Score, SCORE.2=p-Value, SCORE.3=FDR}，本是 SINGLE，
     但模型误判为 PAIR (+SCORE.1/-SCORE.2)，把 p-Value 当成了独立富集列。
   - 否决闸抓住（必需基因 mean_pct = +0.262/+0.312/+0.227）。
   - 处理（用户批准）：用核心必需基因 ground truth 确定性定向。
     这 3 个都是 KO 增殖筛、各含 10 个核心必需基因，方向铁律 = 必需基因必为负。
     apply_directionality.py --anchor-resolve-conflicts：
       重判为 SINGLE，符号取使必需基因落负端的那个 = sign -1。
       结果 ess_mean_pct: 1050 -0.443 / 1052 -0.456 / 1053 -0.367 ✓
     决定写回 overrides.json (resolution="essential-gene-anchor")，保证可复现。

最终方向状态（screen_metadata.SCORE_BASIS）
-----------------------------------------
   64  LLM_SINGLE          (gpt-4o 高置信单列定向)
    4  LLM_PAIR            (gpt-4o 识别的 pos/neg 配对)

  其余 ~2046 个仍是 Phase 1 确定性结果，未改动。

  screen ids: 1081,1082,1083,1441,1442,1443,1444,1445,1446,1447,1448,1697,1698,

[2026-06-21]  assay_domain 分层 —— fitness / stress / reporter
================================================================================

背景
----
- 发现 pooled 统计会把不同表型的屏混在一起，稀释信号。典型：POLA1 是泛必需基因，
  但所有屏一起算 median 只有 -0.75；因为里面混进了药物/病毒/报告屏（POLA1 在那些屏里
  不一定耗竭）。
- 根因不在 harmonization：percentile 是屏内排名，对每个屏都正确。问题在于"跨屏汇总/
  相关时把语义不同的轴混在一起"。harmonization 的符号数学对所有 survival 屏是同一套，
  不需要也不该建多条轴；正确做法是打一个分层标签，下游按标签汇总/相关。

决策
----
  给每个屏打 assay_domain（确定性规则，源自 PHENOTYPE，免费、无 LLM）：
    fitness  — baseline 生长/活力。essentiality 在这里读；连续相关的干净池。
    stress   — 施加压力下的存活（药/病毒/毒素…）。conditional fitness，主要在同压力内可比。
    reporter — 按 marker(FACS)分选，不测存活。每个 marker 自成一轴 → 排除出定量轴和相关
               网络，只用于 per-gene 功能注解。
    other    — 兜底（同 reporter 处理）。
  关键：harmonization 不变；标签只控制下游怎么"汇总/相关/展示"。

改动文件
--------
- script/llm_metadata_extractor.py  新增 rule_assay_domain() + 两列
  (assay_domain, confidence_domain)，prompt_version=v4.1-assay-domain
- screen_metadata_curated 表        DROP 重建，全 2157 行回填
- web/app.py                        /api/gene 改成 domain-aware：按 domain 拆分统计，
  fitness 为主、stress 副、reporter 仅计数+功能探针；interpret prompt 也 domain-aware
- web/index.html                    fitness 主视图(轴+直方图+context) / stress 紧凑副视图 /
  reporter 一句话；fitness↔stress 符号相反时显示"翻转"提示

分布(2157 屏)
-------------
  fitness=1227   stress=700   reporter=230

验证(domain 拆开揭示真生物学)
----------------------------
  POLA1: pooled -0.75 → fitness -0.872(干净 essential) / stress -0.258 / reporter +0.417(稀释源)
  KRAS : fitness -0.666(致癌依赖) / stress +0.064(某些药下丢了反而抗药 —— pooled 把这个糊掉)
  TP53 : fitness +0.491(抑癌，loss 有利) / stress +0.186 / reporter ~0

下游规则(建议)
--------------
- essentiality 只从 fitness 读
- correlation：fitness 一个干净池；stress 同压力内；reporter 完全排除(碎：203 屏跨 114 marker)
- reporter 仍保留作精确功能注解(如 "Senescence genes" / "Regulators of mitophagy")

  NCBI summary + STRING 伙伴 chip,可点击跳转该基因);AI 解读显示 PMID 引用链接。

踩坑

[2026-06-22]  方向(sign)反转 bug 全库修复
================================================================================

怎么发现的
----------
  网站上 POLR2A(RNA Pol II，泛必需)在 HL-60/K-562 等 fitness 屏里显示 +1.00
  (knockout favored)——必需基因在活力屏里该是 -1。用核心必需基因 panel 当探针
  全库审计，发现 971 个 fitness-KO 屏里有 39~42 个整屏 sign 反了。

根本原因(不是一个 bug)
----------------------
  directionality / signedness 无法从 SCORE.k_TYPE 名字可靠推出，同名 metric 不同
  作者用相反约定，甚至 signed/unsigned 都不一样：
   - 同名反约定: Dependency score([0,1]概率,高=必需) vs CERES(负=必需);
     Essentiality Score(这批负=必需) ; Mean Depletion ; Z-score(62% 屏反)
   - signed 被当 magnitude 还 clip(lower=0): RSA(全负)/CGI(51%负)/CasTLE Score
     (屏18 signed 屏157 unsigned) —— clip 把必需信号抹成 0
   - ambiguous/错标 selection: 纯显著性屏 sel 猜错
  validate 之前没抓到：它按 ANALYSIS 跨屏平均，少数反转屏被稀释。

修复(两层，确定性，非破坏性)
---------------------------
  harmonize_scores.py registry 修正:
    - DIR_POS 增 "mean depletion","essentiality score"; 去 "dependency score"
    - DIR_NEG 改为 {"bayes factor","dependency score"}
    - SIG_MAG 路径加【数据驱动 signedness】: 列里有实质负值(>5% 且 min<-0.5)→
      判为 SIGNED_MAG，按 signed 用，不 clip(自动处理 RSA/CGI/CasTLE 因屏而异)
  script/fix_directionality.py(新):
    Phase 1 — 重算 30 个 registry 改了类型的屏(从原始文件，就地替换)
    Phase 2 — 必需基因锚点，只对 FITNESS-KO 屏(stress 排除！见下),
              ≥3 个核心必需基因且 panel 均值>0 → 整屏 negate(精确取反),
              SCORE_BASIS 标 [ANCHOR_FLIP]。共翻 32 个。

为什么 stress 不用锚点
---------------------
  stress(尤其正选择抗性屏)的必需基因在处理/对照两组都耗竭、相互抵消，本就近 0
  (分布中心在 0，不是 -1)。拿"必需基因须负"去翻 stress 会制造新错。故只锚 fitness。
  fitness 分布: 885/971 在 <=-0.5(正确)，正端 42 个=真反转。

结果
----
  validate_harmonization: PASS;  fitness-KO panel 仍为正: 42 → 0/971
  POLR2A fitness median -0.985;  POLR2A fitness >0.9: 12 → 3(残留是真实离群,panel 仍负)
  没有过度修复: stress 不动、近零 borderline 不动。
  ⚠ 必需基因 panel 起初只有人源大写符号 → 漏了小鼠(Polr2a)。补 mouse ortholog 后
    再跑:小鼠 fitness-KO 反转 16 → 0。最终 人源 0/971 + 小鼠 0/70 全部达标。
  边界:KO fitness 两物种已锚点验证;stress(必需基因近0,不可锚)+ CRISPRa/i 靠 registry。

遗留 / 待办
----------
  - correlation_analysis 是基于旧(部分反转)分数建的，受影响的 62 个屏的相关需重算
    → 下次重跑 compute_correlations.py(建议先改成 domain-aware)
  - 复现: python3 script/fix_directionality.py  (registry 改动已在 harmonize_scores.py，
    将来全量重建 harmonize 会自动带上；fix_directionality 处理就地修正)
